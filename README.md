# Smart Task Analyst (Antigravity ✕ Local LLM ✕ DB Demo)

本プロジェクトは、Google Antigravityを用いた開発デモであり、ローカルファーストな永続化、超高速ローカル分析、そしてローカルLLMを用いたセキュアなAIデータ分析を組み合わせたスマートなタスク＆作業ログ分析プラットフォームです。

---

## 💡 このデモが解決する課題（ビジネス価値）

1. **機密データのプライバシー保護 (100% Local-First)**
   - 開発ログやソースコード情報などの機密データを外部のクラウドサービスやSaaS型AI（OpenAI等）に送信せず、完全に自身のPC内でセキュアに管理・分析できます。
2. **クラウドAPIコストの削減**
   - ローカルLLM (Ollama) をエンジンとして用いるため、API利用料を一切発生させずに無制限にAIクエリやインサイトレポートを回せます。
3. **データ分析スキルの壁を打破**
   - 複雑なOLAPクエリを書かなくても、自然言語で「先週一番時間を費やした作業は？」と尋ねるだけで、AIが自動でSQLを組み立てて **DuckDB** で実行し、ビジュアルグラフやレポートを提示してくれます。

---

## 🛠 システム構成とDB選定のストーリー

- **SQLite**: 日常のタスクや操作ログを軽量にローカル保存。
- **DuckDB**: SQLiteのデータファイルを直接アタッチしてクエリを投げる（DuckDBのSQLite拡張機能を利用）。ローカルでありながら高速な集計・OLAP分析を担当。
- **Ollama**: 自然言語からのSQL生成（Text-to-SQL）と、集計結果を基にした改善レポート（日本語）の生成。
- **PostgreSQL**: 将来的なチーム共有や本格的なクラウド移行を模擬するための同期先DB。同期ボタン一つでSQLiteの未同期データをバルクでPostgreSQLに転送。

## 🚀 クイックスタート

### 前提条件
1. **Docker / Docker Compose** がインストールされていること。
2. **Ollama** がホストOSにインストールされ、起動していること。
   - [Ollama公式サイト](https://ollama.com/) からインストールできます。
   - 以下のコマンドで、使用するLLMモデルを事前にプルしておきます：
     ```bash
     ollama pull gemma2:9b
     ```
     *(※ 別モデルを使う場合は、`docker-compose.yml` 内の `OLLAMA_MODEL` 環境変数を適宜変更してください。)*

### 起動手順

1. 本リポジトリをクローンします。
2. リポジトリのルートディレクトリで以下のコマンドを実行し、コンテナを起動します：
   ```bash
   docker-compose up --build
   ```
3. 起動後、ブラウザで以下のURLにアクセスします：
   - **フロントエンド (ダッシュボード)**: [http://localhost:3000](http://localhost:3000)
   - **バックエンド (FastAPI APIドキュメント)**: [http://localhost:8000/docs](http://localhost:8000/docs)

## 📈 デモの実演・検証シナリオ

1. **タスクと作業ログの登録**
   - ダッシュボードの左側「Tasks」パネルからタスク（例: `ユーザー認証機能の実装`）を追加します。
   - 中央の「Work Logs」パネルから、そのタスクを選択し、作業時間（分）とカテゴリ（例: `development` / `debugging`）、および説明を入力してログを追加します。
2. **AIデータアナリストに質問**
   - チャットエリアから「作業時間の合計をカテゴリ別に見せて」や「一番開発に時間がかかっているタスクは何か教えて」と自然言語で質問します。
   - AIがSQLを生成し、DuckDB上で実行され、結果がテーブル表示されると同時に、AIによる見解レポート（Markdown）が返却されるのを確認します。
3. **クラウド同期の実行**
   - 右側の「Cloud Sync Status」パネルを確認します。「未同期のタスク・ログ」の件数が表示されています。
   - 「クラウド PostgreSQL へ同期」ボタンをクリックします。
   - 同期が完了すると、PostgreSQLにデータが移行され、SQLite側の同期ステータスが更新されます。
